## Context

当前 Rust `read_dir_recursive` 会同步遍历整个工作区，并通过一次 IPC 返回嵌套 `FileEntry`；前端 `refreshFileTree` 随后清空并重建全部 DOM。watcher 的防抖只能降低刷新频率，不能限制遍历、序列化、队列积压或 DOM 分配。该问题横跨 Tauri 命令、notify 事件处理、设置、前端状态和渲染，且必须保留文件操作、活动文档与展开状态语义。

## Goals / Non-Goals

**Goals:**

- 首屏工作量与根目录规模相关，而非与整个工作区文件总数相关。
- 常规文件事件仅修改最小节点集合，事件风暴的内存占用有硬上限。
- 忽略、分页、恢复和指标采用前后端一致的语义。
- 用可重复基准量化 10k/100k 文件场景，并以数据决定虚拟列表阈值。

**Non-Goals:**

- 不建立通用文件索引、全文搜索数据库或跨会话完整目录缓存。
- 不跟随符号链接，也不放宽现有工作区路径安全边界。
- 不保证 notify 在所有平台产生相同原始事件序列；适配层负责归一化。
- 不在本变更中重构无关的侧边栏文件操作或编辑器生命周期。

## Decisions

### 1. 以浅层分页命令替代递归目录载荷

新增浅层 `read_dir` 请求，输入目录 path、可选游标和 limit，输出直接子项、下一游标及截断状态。条目继续使用规范化绝对路径，并增加“目录是否可能有子项”提示；前端目录状态区分 `unloaded/loading/loaded/stale/error`。

选择游标分页而不是单纯提高条目上限，是为了让单目录几十万文件时仍有确定的 IPC 与渲染上限。稳定排序键由“目录优先 + 平台约定的名称比较 + path”组成，游标编码排序键；若分页期间目录发生变化，则通过 generation/stale 标记重新加载该目录，避免脆弱的偏移量分页。

替代方案是保留递归命令并后台线程化；它只能改善 UI 阻塞，不能解决总分配量和首屏等待，因此不采用。

### 2. 前端维护 path-keyed 规范化树状态

引入独立于 DOM 的节点 map 与父子有序 path 列表。渲染器根据状态创建或修补对应节点，展开时触发浅层请求；创建、删除、重命名和拖拽也通过同一状态更新入口，避免命令成功后调用整树刷新。

绝对规范化 path 是现有 IPC 和文件操作的共同标识，迁移成本低。Windows 比较需统一分隔符并按文件系统语义处理大小写，展示名称仍保留原始形式。

替代方案是以 DOM 作为唯一状态并原地查询修改；它难以保证重命名后代、分页、失效恢复和测试的确定性，因此不采用。

### 3. watcher 在后端归一化、过滤和施加容量限制

notify 原始事件先转为内部 `TreeEvent`（create/modify/remove/rename/rescan），尽早应用与扫描相同的 ignore matcher，再进入固定容量 channel。短合并窗口按 path 与 kind 折叠重复修改、抵消安全的 create/remove 组合，并保留 rename 的 from/to 关联与批次边界。

容量耗尽时不阻塞 notify 回调，也不创建额外无界任务；设置 overflow 标志并安排一个去重的 rescan 事件。消费者优先将可解释事件批量发送前端，恢复范围优先选择最小已知祖先。

替代方案是在前端接收全部原始 notify 事件后合并；这仍会扩大 Rust→IPC 流量，且无法为生产队列提供 backpressure，因此不采用。

### 4. 忽略规则由设置生成单一快照

默认目录名规则为 `.git`、`node_modules`、`target`、`dist`。设置保存用户追加/排除模式，Rust 编译为 matcher 快照，目录读取与 watcher 共享该快照；前端仅负责编辑配置和在配置变更后使受影响节点失效。

规则变更采用重建 watcher/matcher 并受控同步已加载范围，而不是尝试回放被忽略期间的事件。无效模式在保存时报告，保留最后一个有效快照。

### 5. 恢复优先局部失效，工作区重扫作为最后手段

前端对未加载目录的事件只设置 stale；对已加载目录增量应用。如果父节点缺失、rename 无法配对或 generation 不一致，则重新读取最小共同已加载祖先。只有无法确定范围或根级 overflow 才重读根目录；仍不递归加载折叠目录。

### 6. 指标与基准先行，虚拟化由阈值触发

Rust 使用结构化 tracing 记录扫描和队列指标，前端记录增量批次、局部/根刷新和挂载节点数。基准生成固定种子的目录夹具，并分离测量浅层读取、事件归一化/合并和前端状态更新。

先建立当前与新实现对照，再确定虚拟列表启用阈值。若普通分页和懒加载已使可见 DOM 保持低于阈值，则保留非虚拟渲染；超过阈值时采用扁平化可见行模型进行窗口化，避免预先绑定具体库。

## Risks / Trade-offs

- [分页期间目录变化导致遗漏或重复] → 使用稳定游标与 generation 校验；不一致时使目录失效并重读。
- [跨平台 rename 事件形态不同] → 用平台无关归一化状态机覆盖 from/to/both/other，并用夹具测试典型序列。
- [忽略规则让用户误以为文件丢失] → 设置界面展示默认规则和覆盖项，并为被忽略目录提供明确说明。
- [局部状态模型增加复杂度] → 将 reducer 设计为纯函数，集中 path rewrite、排序和不变量检查并提供单元测试。
- [队列溢出造成短暂陈旧] → 记录 overflow、合并恢复请求，并以最小范围重扫达成最终一致。
- [虚拟化影响拖拽和键盘导航] → 以基准门槛分阶段启用，并用可见行模型与端到端交互测试保护行为。

## Migration Plan

1. 建立基准与指标，记录递归扫描/整树刷新基线。
2. 增加浅层分页命令和共享忽略 matcher，同时保留旧递归命令作为短期兼容路径。
3. 引入 path-keyed 前端状态与懒加载渲染，将初始化和展开切换到新命令。
4. 增加归一化、有界 watcher 管线与增量事件消费，逐个迁移文件操作的整树刷新调用。
5. 启用溢出/不一致恢复，删除生产路径上的递归刷新；确认无调用后移除旧命令。
6. 在基准达到门槛时启用窗口化，否则记录不引入虚拟化的测量结论。

回滚时可在旧命令仍保留期间通过内部开关恢复递归加载；数据格式仅涉及可丢弃的 UI 状态和新增设置默认值，不需要用户数据迁移。

## Open Questions

- 默认单页条目上限、队列容量和事件合并窗口应由基准确定，初始候选分别为 500、2048 和 100–200ms。
- 用户忽略规则采用 glob 还是 gitignore 语义，需要结合现有设置 UI 与依赖体积选择；两端必须共享同一 Rust matcher 结果。
- 虚拟列表是否需要引入依赖取决于可见行基准；优先评估轻量自实现窗口化与现有拖拽兼容性。
